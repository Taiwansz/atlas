## Introduction

This document details the **8 Core User Journeys** of the Atlas Engineering OS. These journeys map user interactions across different engineering personas (defined in [PERSONAS.md](PERSONAS.md)) to systems, CLI commands, and web dashboard touchpoints.

---

## Journey 1: Onboarding & Project Initialization

- **Primary Persona:** *Marcus (Principal Architect) & Sarah (Tech Lead)*
- **Trigger:** Initiating a new microservice repository.
- **Workflow:**
  1. Sarah runs `agy init billing-service --domain fintech` in the terminal.
  2. The CLI prompts for structural preferences (Monorepo vs. Standalone, Relational vs. Graph DB).
  3. Sarah connects the project to the organization via `agy login`.
  4. The Constitution Engine generates a base `atlas.constitution.yaml` populated with regional regulatory compliance defaults (GDPR and PCI-DSS, due to the `fintech` domain choice).
  5. The CLI creates a base `atlas.blueprint.yaml` containing the initialized topology.
- **Output:** Initialized `.atlas` configuration with approved base constitution and blueprint.

---

## Journey 2: Socratic Feature Discovery

- **Primary Persona:** *Elena (Product Manager) & Sarah (Tech Lead)*
- **Trigger:** Initiating a new payment orchestration system.
- **Workflow:**
  1. Elena runs `agy discover --feature "Stripe Multi-Payouts"`.
  2. The Intake Engine starts a terminal-based Socratic interview, prompting Elena for:
     - Expected transactions per second.
     - Payout settlement target windows (e.g., immediate vs. batch T+2).
     - Data residency requirements (e.g., European payouts must stay in EU databases).
  3. The agent notices an ambiguity regarding retry failures and asks: *"If Stripe API is down during payout commit, do we enqueue backoff or fail immediately with manual override?"* Elena inputs: *"Enqueue retry for up to 3 attempts, then trigger webhook to customer support."*
  4. The engine records this requirement context and compiles `.atlas/requirements/stripe-payouts.json`.
- **Output:** Structured requirement JSON file mapping functional and regulatory constraints.

---

## Journey 3: Blueprint Refinement & ADR Approval

- **Primary Persona:** *Marcus (Principal Architect)*
- **Trigger:** Reviewing and approving the architecture generated from the discovery intake.
- **Workflow:**
  1. Marcus receives a slack notification generated by the webhook: `Blueprint draft proposed for Stripe Multi-Payouts`.
  2. Marcus logs into the Atlas Web Dashboard and navigates to the visual Blueprint Editor.
  3. He views the proposed changes: a new service component `PayoutProcessor` implementing a Stripe adapter interface, storing transaction state in Postgres, and publishing events to Kafka.
  4. He notices the agent proposed using HTTP pooling for Stripe queries. He overrides this in the UI, requiring gRPC communication to the internal user-context service.
  5. He approves the resulting auto-generated Architectural Decision Record (`ADR-013-stripe-payout-orchestrator.md`).
- **Output:** Approved and locked `atlas.blueprint.yaml` and committed ADR.

## Journey 4: Multi-Agent Implementation Execution

- **Primary Persona:** *Alex (Mid-level Developer)*
- **Trigger:** Applying the approved architecture changes to write actual source code.
- **Workflow:**
  1. Alex pulls the latest blueprint and runs `agy orchestrate --apply --watch` on the branch.
  2. The Agent Orchestrator spins up:
     - **Database Agent:** Generates SQL migrations for the transaction ledger.
     - **Contract Agent:** Generates interface file definitions in TypeScript.
     - **Implementation Agent:** Generates implementation source code file logic.
  3. Alex watches the log stream displaying tool calls, file writes, and code test runs.
  4. A unit test fails because Stripe mock responses are malformed. The QA Agent automatically refactors the mock files, runs the test suite again, and passes the build.
- **Output:** Validated source code, database migrations, and unit tests.

## Journey 5: Code-Drift Auditing & Scoring

- **Primary Persona:** *Sarah (Tech Lead) & Marcus (Principal Architect)*
- **Trigger:** Pre-merge CI pipeline check or local quality analysis.
- **Workflow:**
  1. Alex submits a Pull Request.
  2. The CI pipeline runs `agy audit`.
  3. The Audit Engine parses the codebase AST and compares it against the locked blueprint.
  4. **Drift Detected:** Alex added a custom database connector function directly in the billing package without declaring it in the blueprint component contracts.
  5. The build fails with exit code `2` (Syntax/Schema Validation Error), blocking the merge.
  6. The PR dashboard displays the drift warning: `Code Drift: Undocumented endpoint route '/payouts/test' found in route.ts`.
- **Output:** Detailed build audit report blocking the pipeline and maintaining repository integrity.

## Journey 6: Red Team Simulation & Threat Mitigation

- **Primary Persona:** *Vikram (Security Engineer)*
- **Trigger:** Nightly security audit trigger.
- **Workflow:**
  1. The Red Team Engine initiates a simulation job.
  2. It scans the proposed blueprint and AST code, discovering that the `PayoutProcessor` accepts client-provided redirects without validating the domain.
  3. It executes a simulated open-redirect attack, verifying the vulnerability.
  4. It logs a high-severity security finding: `Open Redirect Vulnerability in Payout Router`.
  5. It automatically generates a mitigation pull request containing updated routing checks.
  6. Vikram reviews the security finding on the dashboard and approves the fix PR.
- **Output:** Mitigated vulnerability and updated secure blueprint profile.

---

## Journey 7: Continuous System Refactoring / Evolution

- **Primary Persona:** *Sarah (Tech Lead)*
- **Trigger:** Refactoring a legacy auth system to support Keycloak SSO.
- **Workflow:**
  1. Sarah runs `agy evolve --from auth-v1 --to keycloak-sso`.
  2. The Evolution Engine reads the legacy auth schemas and outputs a migration roadmap.
  3. The Agent Orchestrator updates dependencies, shifts the passport middleware imports to OidcClient wrappers, and creates a data migration blueprint.
  4. Sarah reviews the proposed changes locally and runs `agy orchestrate --apply`.
- **Output:** Upgraded components conforming to new architectural patterns.

---

## Journey 8: Custom Tool & MCP Server Integration

- **Primary Persona:** *Vikram (Security Engineer)*
- **Trigger:** Adding a proprietary code auditing engine to Atlas.
- **Workflow:**
  1. Vikram runs `agy mcp link --command "docker run -i mycorp/security-mcp"`.
  2. The MCP Discovery Engine queries the container capabilities, registering standard security tools.
  3. The tools are mapped into the local `.atlas/mcp-config.json`.
  4. During subsequent audits, the Agent Orchestrator utilizes the custom container to perform security analysis.
- **Output:** Configured MCP integration and custom security capabilities.
