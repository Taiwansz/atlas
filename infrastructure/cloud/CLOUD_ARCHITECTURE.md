## 1. Global Network & Region Architecture

Atlas is deployed as a cloud-neutral, multi-region platform. The primary deployment targets are AWS and GCP. In high-security enterprise environments, Atlas can be deployed 100% on-premises.

```
                         [Global CDN / Cloudflare]
                                    │
                                    ▼ (HTTPS / WSS)
                 [VPC Ingress Gateway / Load Balancer]
                                    │
       ┌────────────────────────────┴────────────────────────────┐
       ▼ (Public Subnet)                                         ▼ (Private Subnet)
[Next.js App / CDN Cache]                               [gRPC API Gateway]
                                                                 │
                                                                 ▼
                                                        [EKS Cluster Nodes]
                                                        (Engines & Agents)
```

## 2. Cluster Networking & Security Groups

1. **VPC Layout:** Spans three availability zones (AZs) containing separate public, private, and isolated database subnets.
2. **Private Nodes:** The EKS (Elastic Kubernetes Service) worker nodes run exclusively in the private subnets. They communicate with the public internet via NAT Gateways only.
3. **Network Policies:** Pod-to-pod communication is restricted by namespace. For example, the `web-dashboard` service cannot talk to database pods directly; it must interface with the `api-gateway` service.
4. **Ingress Control:** Traefik or NGINX Ingress Controller acts as the load-balancer target, decrypting TLS certificates and forwarding requests.

## 3. Data Storage & Replication Strategy

To scale and prevent data loss under high concurrent read/write loads:

### 3.1 Relational Store (PostgreSQL)
- **Engine:** AWS RDS Aurora PostgreSQL (Multi-AZ).
- **Replication:** One primary writer instance, with three read-replicas scaled dynamically across AZs. Write operations target the writer, while read queries (via the dashboard API) route to the reader endpoint.

### 3.2 Knowledge Graph (Neo4j)
- **Deployment:** Neo4j Causal Cluster containing three Core servers and two Read Replicas.
- **Invariants:** Core servers participate in Raft consensus to guarantee write transactions and ACID compliance. Read replicas handle heavy query routing.

### 3.3 Vector Indexing (Qdrant)
- **Deployment:** Distributed Qdrant cluster running on statefulsets with local NVMe drives.
- **Replication:** Configured with replication factor `2` to prevent data loss if a Kubernetes node fails.

## 4. Sandbox Execution Isolation (gVisor & Firecracker)

Atlas runs code generated by AI agents to verify tests and run simulations. Because this code is untrusted, executing it directly on host cluster nodes is a severe security risk.

- **Sandbox Layer:** All generated code verification runs inside ephemeral Kubernetes pods using the **gVisor** container runtime (`runsc`). gVisor intercepts syscalls to prevent container escapes.
- **Heavy Simulations:** Resource-heavy system simulation runs are executed inside ephemeral **Firecracker microVMs** managed by an AWS EC2 instance pool. MicroVMs are created and destroyed in milliseconds, providing hardware-level isolation.
